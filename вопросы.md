# Питання клієнта — відповіді

---

**1. Наскільки я зрозумів, всі зміни для торгівлі (або майже всі) я, як користувач бота, робитиму у файлі config/config.yaml. Для прикладу, я зміню параметри трейлінг-стопу, чи ще якісь у цьому файлі. Що потрібно зробити, щоб ці зміни збереглися і в подальшому використовувалися як робочі параметри при відкритті нових угод на торгових парах? Напишіть будь ласка, детально, які дії потрібно виконати для збереження оновлених параметрів.**

**Відповідь:** Так, основні налаштування змінюються в `config/config.yaml`. Щоб зміни почали працювати на Render:

1. Відкрийте файл `config/config.yaml` у редакторі (VS Code тощо).
2. Внесіть потрібні зміни (наприклад, `trailing_stop_pct: 1.0`).
3. Збережіть файл (Ctrl+S).
4. Відкрийте термінал у папці проекту і виконайте:
   - `git add config/config.yaml` — додати файл до змін
   - `git commit -m "Оновлення трейлінг-стопу"` — зафіксувати зміни
   - `git push origin main` — відправити на GitHub
5. Render автоматично побачить зміни в репозиторії і перезапустить бота з новими параметрами (зазвичай 2–5 хвилин).

Після цього нові угоди вже відкриватимуться з оновленими параметрами.

---

**2. Що відбудеться, якщо вже відкриті (діючі) угоди на декількох торгових парах, і у цей час змінити якісь показники (наприклад, той же трейлінг-стоп), чи додати нові торгові пари чи ще щось? Які будуть наслідки? Чи змінювати параметри при хоча б одній діючій угоді можна взагалі?**

**Відповідь:** Змінювати параметри можна в будь-який момент, навіть коли є відкриті угоди.

- **Відкриті угоди** — залишаються з тими умовами, з якими були відкриті. Трейлінг-стоп і інші параметри застосовуються лише до **нових** угод.
- **Нові угоди** — відкриваються вже з оновленими параметрами з конфігу.
- **Додавання нових пар** — нові пари з’являться в Telegram після деплою; їх можна запустити кнопкою ▶. Це не впливає на вже відкриті позиції по інших парах.
- **Зміна трейлінг-стопу** — поточні позиції продовжують працювати зі старим трейлінгом; нові позиції будуть з новим.

**3. Як бот чи його процеси захищені від стороннього втручання (за допомогою паролів, чи якось інакше)? Як зробити так, щоб бот був максимально безпечним від хакерів та тому подібних загроз? Чи достатні засоби вже зараз для цього задіяні на сервері Render та на GitHub?**

**Відповідь:** Захист побудований так:

- **API-ключі та токени** зберігаються в змінних оточення Render (Environment Variables), не в коді. У репозиторії їх немає.
- **GitHub** — доступ до репозиторія лише у вас (або у тих, кому ви дали доступ). Репозиторій можна зробити приватним.
- **Telegram** — керувати ботом може лише той, хто знає вашого бота. Рекомендація: не публікувати посилання на бота і не ділитися скріншотами з командами.
- **Додаткові кроки:**
  - У Bybit: створіть API-ключі з обмеженими правами (лише торгівля, без виведення коштів).
  - Увімкніть IP whitelist для API-ключів Bybit (якщо це підтримується).
  - Регулярно перевіряйте активність на Bybit і в Telegram.

Render і GitHub дають базовий захист; найважливіше — не розголошувати API-ключі та токен бота.

---

**4. Що може відбутися при втраті інтернет з'єднання у мене, або на сервері, або на GitHub? Це може і малоймовірно, але я в Україні і звик ще й не до такого трешу.**

**Відповідь:**

- **У вас вдома** — нічого з ботом не станеться. Він працює на Render, не на вашому комп’ютері.
- **На сервері Render** — якщо Render втратить інтернет, бот тимчасово не зможе робити запити до Bybit і Telegram. Позиції на біржі залишаються відкритими. Після відновлення зв’язку бот продовжить роботу. Трейлінг-стоп і стоп-лоси виконуються ботом, а не біржею, тому під час простою вони не спрацюють.
- **GitHub** — втрата доступу до GitHub не впливає на вже запущений бота на Render. Впливає лише на можливість деплою нових змін.

**5. У такому випадку краще за все спробувати якось зупинити бота?**

**Відповідь:** Якщо очікується тривала відсутність інтернету на Render або ви хочете припинити торгівлю:

1. **Через Telegram** — натисніть «Зупинити все» або ⏸ біля кожної пари. Бот перестане відкривати нові позиції (якщо є доступ до інтернету).
2. **Через Bybit** — закрийте позиції вручну в інтерфейсі біржі (див. питання 7).
3. **Через Render Dashboard** — можна зупинити сервіс (Suspend), тоді бот повністю перестане працювати.

Якщо інтернету немає, Telegram і Render будуть недоступні. Тому має сенс мати доступ до Bybit (з телефону або іншого пристрою), щоб при потребі закрити позиції вручну.

**6. Де бот буде відображатися на біржі Bybit – на сторінці ф’ючерсних торгів? У якому розділі ф’ючерсів - Позиції, Активи, Відкриті ордери чи Інструменти?**

**Відповідь:** Бот торгує через API Bybit, тому його угоди видно в звичайному інтерфейсі біржі:

- **Розділ:** Derivatives → USDT Perpetual (або «Ф’ючерси» → «USDT Perpetual»).
- **Позиції** — тут відображаються відкриті позиції бота (ETHUSDT, BTCUSDT тощо).
- **Відкриті ордери** — тут будуть лімітні ордери бота (якщо вони використовуються).
- **Активи** — загальний баланс, маржа, PnL.
- **Інструменти** — вибір торгової пари.

Бот не має окремого інтерфейсу на Bybit; він просто виконує ордери через API, і вони з’являються в цих розділах як звичайні угоди.

**7. Чи можна закрити бота з самої біржі Bybit – конкретну діючу угоду чи всі відкриті угоди разом?**

**Відповідь:** Так. З Bybit можна закрити і одну угоду, і всі:

- **Одна позиція:** Позиції → знайдіть потрібну пару (наприклад, ETHUSDT) → кнопка «Close» / «Закрити».
- **Усі позиції:** Є опція «Close All» / «Закрити всі» (залежно від версії інтерфейсу).

Після ручного закриття на біржі бот виявить, що позиції немає, і при наступному сигналі зможе відкрити нову. Ручне закриття не вимикає бота — він продовжує працювати і може відкривати нові угоди.

**8. Поясніть детально, будь ласка, що у даному випадку означають наступні дії (текст з Детального гайду):**

**Крок 3: Деплой змін на Render**

**Закомітьте та запуште:**
```
git add config/config.yaml
git commit -m "Update pairs and max_usdt"
git push origin main
```

**Що мається на увазі у п. 2-4 (бажано пояснити детально, кроково)? Що означають ці терміни у п. 2? І де саме знаходиться цей текст/файл?**

**Відповідь:**

Це команди для **Git** — системи контролю версій, яка зберігає зміни в коді і конфігурації.

- **`git add config/config.yaml`** — позначити файл `config/config.yaml` для збереження. Git починає відстежувати зміни саме в цьому файлі.
- **`git commit -m "Update pairs and max_usdt"`** — створити «коміт» (знімок змін) з коротким описом. Текст у лапках — це ваше повідомлення про зміни (можна писати будь-яке).
- **`git push origin main`** — відправити збережені зміни на GitHub у гілку `main`. Після цього Render бачить оновлення і запускає новий деплой.

**Де виконувати:** у терміналі (Command Prompt, PowerShell або Terminal у Cursor) у папці проекту `torgovyibot`. Спочатку перейдіть у папку: `cd C:\Users\...\Desktop\torgovyibot`, потім виконайте команди.

**Де знаходиться файл:** `config/config.yaml` — у папці проекту, у підпапці `config`.

**9. У розділі «Що відбувається далі» Детального гайду:**

**«Кожні ~60 секунд бот перевіряє сигнали для кожної активної пари» — чи не завершився період між запитами, як ви вважаєте? Якщо його змінити, умовно до 30 секунд, на що це вплине? Чи якось відобразиться на роботі сервера чи біржі, чи буде потрібно переходити на інший тариф, наприклад? Якщо цього часу достатньо, то міняти у боті нічого не треба.**

**Відповідь:** 60 секунд для таймфрейму 1h достатньо. Свічка 1h оновлюється раз на годину, тому частіші перевірки майже не дають нової інформації.

Якщо змінити інтервал на 30 секунд:
- **Плюси:** трохи швидша реакція на зміни ціни (корисно для 15m).
- **Мінуси:** вдвічі більше запитів до Bybit і Telegram — вище навантаження і ризик спрацювання rate limits.
- **Тариф Render:** зміна інтервалу не вимагає зміни тарифу.

**Висновок:** для 1h залишати 60 секунд доцільно. Змінювати не обов’язково.

**10. У розділі «Важливі нюанси» Детального гайду:**

**«Лімітні входи — можуть не виконатися, якщо ціна пішла далі; для швидких ринків можна виставити use_limit_orders: false.»**

**Це абсолютно критична інформація для торгівлі на ринку крипти. Треба обов’язково змінити абсолютно всі лімітні ордери на ринкові, зважаючи на те, що є певні затримки між запитами (приблизно 60 секунд). Я вас прошу це зробити — внести зміни у код бота.**

**Відповідь:** Зміни внесено. У `config/config.yaml` тепер за замовчуванням встановлено `use_limit_orders: false` — усі входи та виходи по сигналу виконуються ринковими ордерами. Трейлінг-стоп і раніше був лише ринковим.

У конфігу:
```yaml
execution:
  use_limit_orders: false  # ринкові ордери для входу/виходу
```

Якщо потрібно повернути лімітні ордери, можна змінити на `true`.
